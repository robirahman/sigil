Firebase Setup Guide

How to configure Firebase Realtime Database for Sigil online multiplayer

Overview

Sigil's online multiplayer runs entirely in the browser. There is no game server — both players run the game engine locally, and their moves are synchronized through Firebase Realtime Database.

Firebase's free Spark plan provides 1 GB storage and 10 GB/month of data transfer, which is far more than enough for Sigil games (each game uses roughly 5–20 KB of data).

You have two options for setup:

  • Site owner setup — configure Firebase once and hardcode the config so all visitors can play without any setup. This is the recommended approach.
  • Per-user setup — each player pastes their own Firebase config into the multiplayer lobby. This is the current default, requiring no server-side changes.

Option A: Site Owner Setup (Recommended)

If you're deploying this site, follow these steps once to enable multiplayer for everyone.

1 Create a Firebase Project

  1. Go to console.firebase.google.com
  2. Click Add project
  3. Name it (e.g., sigil-online)
  4. Google Analytics is optional — you can disable it
  5. Click Create project

2 Enable Realtime Database

  1. In the Firebase console sidebar, click Build → Realtime Database
  2. Click Create Database
  3. Choose a location (pick the one closest to most of your players)
  4. Select Start in test mode (we'll add proper security rules in step 4)
  5. Click Enable

3 Get Your Web App Config

  1. In the Firebase console, click the gear icon → Project settings
  2. Scroll down to Your apps
  3. Click the </> (Web) icon to register a web app
  4. Name it anything (e.g., "Sigil Web")
  5. You do not need Firebase Hosting — uncheck it
  6. Click Register app
  7. Copy the firebaseConfig object. It looks like this:
{
  "apiKey": "AIzaSyD...",
  "authDomain": "sigil-online.firebaseapp.com",
  "databaseURL": "https://sigil-online-default-rtdb.firebaseio.com",
  "projectId": "sigil-online",
  "storageBucket": "sigil-online.appspot.com",
  "messagingSenderId": "123456789",
  "appId": "1:123456789:web:abc123"
}

4 Set Security Rules

In the Firebase console, go to Realtime Database → Rules and replace the default rules with: 

{
  "rules": {
    "rooms": {
      "$roomId": {
        ".read": true,
        ".write": true,
        ".validate": "newData.hasChildren(['spellNames', 'status', 'created'])",
        "turns": {
          "$turnId": {
            ".validate": "newData.hasChildren(['color', 'actions', 'timestamp'])"
          }
        }
      }
    },
    "completed_games": {
      ".read": false,
      ".write": true,
      "$gameId": {
        ".validate": "newData.hasChildren(['spellNames', 'winner', 'turns', 'timestamp'])"
      }
    },
    ".read": false,
    ".write": false
  }
}

These rules:

  • Allow anyone to read and write game rooms (no authentication required)
  • Require room data to include spellNames, status, and created fields
  • Require each turn record to include color, actions (array), and timestamp
  • Allow writing completed games to /completed_games (write-only — only the site owner can read them via the Firebase console or Admin SDK)
  • Block all other access
Note: These are permissive rules suitable for a game with no user accounts. Anyone with the room code can join. For production use with abuse concerns, consider adding Anonymous Authentication and rate limiting.

5 Hardcode the Config in Your Site

To skip the per-user config step, edit multiplayer.html and add a <script> block that auto-initializes Firebase before Alpine loads. Find this section near the bottom of the file: 

<!-- Firebase SDK -->
<script src="https://www.gstatic.com/firebasejs/8.10.1/firebase-app.js"></script>
<script src="https://www.gstatic.com/firebasejs/8.10.1/firebase-database.js"></script>

Add this immediately after those two script tags:

<script>
  firebase.initializeApp({
    apiKey: "YOUR_API_KEY",
    authDomain: "YOUR_PROJECT.firebaseapp.com",
    databaseURL: "https://YOUR_PROJECT-default-rtdb.firebaseio.com",
    projectId: "YOUR_PROJECT",
    storageBucket: "YOUR_PROJECT.appspot.com",
    messagingSenderId: "YOUR_SENDER_ID",
    appId: "YOUR_APP_ID"
  });
</script>

Then, in the multiplayerLobby Alpine component's init() method, change the initial phase from 'setup' to 'lobby' and initialize the sync object directly: 

init() {
  // Skip config UI — Firebase is pre-configured
  this.sync = new FirebaseSync(firebase.database());
  this.phase = 'lobby';
},

6 Training Data from Human Games

Completed multiplayer games are automatically saved to /completed_games in Firebase. Each record includes the spell list, SFN board snapshots before and after every turn, and the winner. This data can be downloaded and converted into training examples for the AI.

To export human games as training data:

# From the project root directory:
python -m ai.import_human_games \
  --db-url https://YOUR_PROJECT-default-rtdb.firebaseio.com \
  --output ai/data/human_games.jsonl

This replays each game position, enumerates all legal turns, finds the one the human chose (by comparing the before/after SFN snapshots), and writes a training example with policy = 1.0 for that turn. The output JSONL format is identical to self-play data, so you can train on it directly: 

# Train on human games (behavioral cloning):
python -m ai.train_sigil --data ai/data/human_games.jsonl

# Or combine with self-play data:
cat ai/data/selfplay_*.jsonl ai/data/human_games.jsonl > ai/data/combined.jsonl
python -m ai.train_sigil --data ai/data/combined.jsonl
Note: The security rules set /completed_games as write-only (players can write but not read). To download games, either temporarily enable read access in the Firebase console, or use a service account key with the --service-account flag.

7 (Optional) Room Cleanup

Game rooms in /rooms accumulate over time. You can delete old ones manually from the Firebase console, or set up a scheduled Cloud Function: 

// Cloud Function (Node.js)
const functions = require('firebase-functions');
const admin = require('firebase-admin');
admin.initializeApp();

exports.cleanupRooms = functions.pubsub
  .schedule('every 6 hours')
  .onRun(async () => {
    const cutoff = Date.now() - 24 * 60 * 60 * 1000;
    const db = admin.database();
    const snap = await db.ref('rooms')
      .orderByChild('created')
      .endAt(cutoff)
      .once('value');
    const updates = {};
    snap.forEach(child => { updates[child.key] = null; });
    if (Object.keys(updates).length > 0) {
      await db.ref('rooms').update(updates);
      console.log(`Deleted ${Object.keys(updates).length} old rooms`);
    }
  });
Note: Cloud Functions require the Blaze (pay-as-you-go) plan, but the free tier usage should keep costs at $0 for low traffic.

Option B: Per-User Setup

This is the default behavior. Each player creates their own Firebase project and pastes the config JSON into the multiplayer lobby. The config is saved in localStorage so they only need to do it once per browser.

Players should follow steps 1–4 above. Instead of step 5, they paste their config JSON into the text box on the multiplayer page.

Limitation: Both players must use the same Firebase project to connect. If Player 1 creates a room using their Firebase project, Player 2 must also have that same project's config pasted in. In practice, one player should share their config with the other.

Database Structure

Each game room is stored at /rooms/{ROOM_CODE} with this structure:

PathTypeDescription
spellNamesArray[9]The 9 spell names for this game
statusString"waiting", "playing", or "finished"
createdNumberTimestamp (ms since epoch) when the room was created
red/connectedBooleanWhether Red player is currently connected
blue/connectedBooleanWhether Blue player is currently connected
turns/{pushId}ObjectEach completed turn, in order
turns/{pushId}/colorString"red" or "blue"
turns/{pushId}/actionsArrayOrdered list of action strings for this turn (node names, spell names, "pass", "dash", etc.)
turns/{pushId}/timestampNumberWhen the turn was finalized

How Sync Works

The multiplayer system uses a simple protocol:

  1. Room creation: Player 1 generates a random 6-character room code, writes the initial room data (spell names, status) to Firebase, and starts listening for turns.
  2. Room joining: Player 2 enters the code, reads the room data to get the spell names, marks themselves as connected, and starts listening for turns.
  3. Gameplay: Both players run the full game engine locally. When it's your turn, your actions are applied locally but buffered — they are not sent to Firebase until the turn is finalized (End Turn / Pass). This means Reset Turn works correctly without affecting the opponent's state. When it's the opponent's turn, the game engine waits for their completed turn to arrive from Firebase and replays the actions locally.
  4. Turn format: Each turn is sent as an ordered array of action strings: node names ("a2", "b7"), spell names ("Flourish"), or commands ("pass", "dash"). The opponent's engine consumes these one at a time in order.
  5. Disconnect detection: Each player sets an onDisconnect handler that marks them as disconnected when their browser closes.

Troubleshooting

Errors when connecting

  • "Firebase SDK not loaded" — Check your internet connection. The Firebase SDK is loaded from Google's CDN.
  • "Config must include databaseURL" — Make sure you copied the full config object, including the databaseURL field.
  • "Permission denied" — Your security rules may be too restrictive. Check step 4 above.

Room not found

  • Room codes are case-insensitive (auto-uppercased). Double-check the code.
  • Rooms are only valid while at least one player is connected. If both players disconnect, the room data persists in Firebase but the game cannot be resumed.
  • If you set up room cleanup, rooms older than 24 hours will be deleted.

Moves not syncing

  • Both players must be using the same Firebase project. If your configs point to different databases, actions won't be shared.
  • Check the browser console for Firebase errors.
  • Verify the Realtime Database is in the correct region and accessible.
